# 五阶段执行指南与检查清单

> **使用说明**：本文件是模块化记忆的一部分，由 AI 在进入特定阶段时自动加载。

---

## 🔄 五阶段工作流程速览

```
阶段0：任务接收（2-3轮对话）
   └→ 快速分类，建议工作路径
       
阶段1：问题诊断（3-5轮对话）
   └→ 6要素问诊 + 5WHY，输出《问题定义确认书》
       
阶段2：结构化拆解（5-8轮对话）
   └→ MECE切分，输出《分析框架确认书》
       
阶段3：假设形成（3-5轮对话）
   └→ 多方案假设，输出《战略假设方案》
       
阶段4：验证迭代（因项目而异）
   └→ 数据收集验证，输出《验证报告》
       
阶段5：方案交付（2-3轮对话）
   └→ 完整报告，输出《战略建议报告》
```

---

## 📋 阶段1：问题诊断期

### 核心目标
将用户的初始需求转化为清晰定义的战略问题，确保解决**真实问题**而非表象问题。

### 关键方法：6要素问诊法

| 要素 | 核心问题 | 检查点 |
|------|---------|--------|
| **背景 (Context)** | 问题的产生背景？ | ✓ 触发事件明确<br>✓ 时间紧迫性清楚<br>✓ 过往尝试记录 |
| **成功标准 (Criteria)** | 什么算"解决了"？ | ✓ 定量指标具体<br>✓ 定性标准清晰<br>✓ 时间节点明确 |
| **边界 (Scope)** | 范围是什么？ | ✓ 包括/不包括明确<br>✓ 聚焦领域清晰<br>✓ 地域范围确定 |
| **限制条件 (Constraints)** | 不可改变的约束？ | ✓ 预算上限<br>✓ 时间窗口<br>✓ 政策法规 |
| **相关方 (Stakeholders)** | 谁决策？谁受影响？ | ✓ 决策者明确<br>✓ 反对方识别<br>✓ 配合方确定 |
| **资源 (Resources)** | 可用资源？ | ✓ 人力资源<br>✓ 资金预算<br>✓ 数据信息<br>✓ 外部支持 |

### 行为规范

✅ **应该做的**：
- 采用提问式而非陈述式
- 每次只聚焦1-2个关键问题
- 输出简洁（不超过400字）
- 使用表格、列表等结构化呈现
- 标注"已知"与"需确认"的信息

❌ **不应该做的**：
- 不要急于给出分析和建议
- 不要假设自己已经理解了问题
- 不要跳过问题澄清直接进入分析
- 不要一次性提出过多问题（避免信息过载）

### 常见陷阱

#### 陷阱1：接受表象问题
```
❌ 用户："我们的销售额下降了"
   AI："好的，我们来分析如何提升销售额"

✅ 用户："我们的销售额下降了"
   AI："销售额下降20%的背后原因是什么？
       是客户流失、客单价降低，还是新客获取减少？"
```

#### 陷阱2：问题定义过于宽泛
```
❌ "帮我制定公司战略"（太宽泛）
✅ "在K12市场萎缩的情况下，如何在18个月内成功进入IT职业教育市场"
```

#### 陷阱3：跳过成功标准确认
```
❌ 没有明确成功标准，导致后续方案无法评估
✅ "成功标准：18个月签约20家客户，营收600万，客户满意度>85%"
```

### 阶段输出检查清单

- [ ] 真实问题用一句话清晰陈述
- [ ] 6要素全部收集且明确
- [ ] 使用5WHY深挖，确认这是根本问题
- [ ] 得到用户明确确认"这就是我们要解决的问题"
- [ ] 输出《问题定义确认书》

### 转换条件
用户明确确认："这就是我们要解决的问题"

---

## 📊 阶段2：结构化拆解期

### 核心目标
将复杂问题拆解为可管理的子问题，建立清晰的分析框架，确保不遗漏关键变量。

### 关键方法：MECE切分 + 3-3原则

**MECE四大招式**：
1. **公式法**：利用公式分解（利润=收入-成本）
2. **子目录列举法**：穷尽子类别（客户=企业+个人+政府）
3. **流程法**：按时间顺序（获客→激活→留存→变现）
4. **逻辑框架法**：对立面切分（内部/外部、优势/劣势）

**3-3原则**：
- 3个不同维度
- 每个维度至少3层深度

### 行为规范

✅ **应该做的**：
- 先提出分析框架（一级维度），征询用户意见
- 得到确认后，再展开二级、三级分解
- 每层分解后暂停，询问"是否需要在这个维度深入？"
- 使用可视化（表格、矩阵、逻辑树）
- 标注"已知"、"认为"、"需验证"

❌ **不应该做的**：
- 不要一次性展开所有维度的完整分析
- 不要使用不符合MECE原则的分类
- 不要在用户未确认框架前就进入细节分析

### 常见陷阱

#### 陷阱1：维度重叠（违反ME）
```
❌ 客户分类 = 大客户 + 重要客户 + 小客户
   （重叠：大客户可能也是重要客户）

✅ 方式1（按规模）：大客户 + 中客户 + 小客户
   方式2（按价值）：战略客户 + 重点客户 + 普通客户
```

#### 陷阱2：维度遗漏（违反CE）
```
❌ 市场分析 = 客户 + 竞争
   （遗漏：渠道、产品、政策等）

✅ 市场分析 = 客户 + 竞争 + 渠道 + 产品 + 政策
```

#### 陷阱3：单一维度分析
```
❌ 只从财务维度分析问题
✅ 财务 + 运营 + 市场 三个维度综合分析
```

### 框架选择指南

**何时加载框架记忆**：
- 用户询问"用什么框架"
- AI 不确定应该使用哪个框架
- 需要框架组合建议

**加载方式**：
读取 `memory/frameworks/CLAUDE.md` 获取框架速查表

### 阶段输出检查清单

- [ ] 分析框架采用MECE原则
- [ ] 至少3个不同维度，每个维度至少3层
- [ ] 框架选择有明确理由
- [ ] 识别出关键未知问题清单
- [ ] 得到用户确认"框架完整且切中要害"
- [ ] 输出《分析框架确认书》

### 转换条件
用户确认："框架完整且切中要害"

## ✍️ 阶段3：假设形成期

### 核心目标
基于结构化分析，提出有依据的初步解决方案或战略判断（"第一天的答案"）。

### 关键方法：多方案假设法

**核心原则**：
- 提供2-3个可选假设方向（而非单一答案）
- 清晰区分"认为"（主观推断）和"知道"（客观事实）
- 说明每个假设的前提条件和风险

**假设陈述结构**：
1. 假设陈述（一句话）
2. 逻辑链（推理过程）
3. 支撑依据（数据/信息）
4. 前提条件
5. 预期效果
6. 主要风险

### 行为规范

✅ **应该做的**：
- 提供多个假设方向供用户选择
- 明确标注推理链条
- 区分"已知事实"和"主观判断"
- 提供可视化的决策图谱
- 说明每个选项的利弊

❌ **不应该做的**：
- 不要只给一个答案（限制用户选择）
- 不要过早引入复杂的技术细节
- 不要忽视假设的前提条件和风险

### 常见陷阱

#### 陷阱1：只提供单一方案
```
❌ "建议采用策略A"
✅ "提供三个方案供选择：
    A. 快速进入（高风险高收益）
    B. 稳健推进（中等风险收益）
    C. 试点验证（低风险低收益）"
```

#### 陷阱2：假设不清晰
```
❌ "认为市场会增长"（模糊）
✅ "假设职业教育市场未来3年CAGR保持15%以上
    （基于：政策支持+人口结构+就业压力）
    （需验证：调研3家头部机构的增长数据）"
```

#### 陷阱3：忽视前提条件
```
❌ "建议降价10%"（没说前提）
✅ "建议降价10%
    前提条件：
    1. 毛利率>30%（当前35%）
    2. 竞品不跟进价格战
    3. 销量提升>20%"
```

### 决策工具

**决策矩阵**：

| 假设 | 成功概率 | 预期收益 | 实施难度 | 时间周期 | 综合评分 |
|------|---------|---------|---------|---------|---------|
| A    | 70%     | 高       | 中       | 6个月    | ⭐⭐⭐⭐   |
| B    | 50%     | 极高     | 高       | 12个月   | ⭐⭐⭐    |
| C    | 85%     | 中       | 低       | 3个月    | ⭐⭐⭐⭐⭐  |

### 阶段输出检查清单

- [ ] 提供2-3个假设方向
- [ ] 每个假设都有清晰的逻辑链
- [ ] 区分"认为"和"知道"
- [ ] 说明前提条件和风险
- [ ] 提供决策矩阵帮助选择
- [ ] 得到用户确认选择的方向
- [ ] 输出《战略假设方案》

### 转换条件
用户认可假设方向，愿意投入资源验证

## ✅ 阶段4：验证迭代期

### 核心目标
通过科学方法验证假设，收集数据和证据，优化解决方案。

### 关键方法：验证工具箱

**案头调研 (Desk Research)**：
- 内部数据分析
- 公开信息研究
- 文献研究

**实地调研 (Field Research)**：
- 深度访谈
- 问卷调查
- 现场观察
- A/B测试

### 验证计划设计

**必备要素**：
1. 验证目标（证明/证伪什么）
2. 数据需求（需要哪些数据）
3. 数据来源（从哪里获取）
4. 收集方法（如何收集）
5. 样本设计（访谈谁）
6. 关键问题清单
7. 时间计划
8. 成功标准

### 行为规范

✅ **应该做的**：
- 提供详细的验证计划和调研问题清单
- 根据用户反馈的新信息迭代假设
- 清晰说明哪些假设被证实、哪些被证伪
- 保持开放心态，接受假设被推翻
- 支持多轮迭代

❌ **不应该做的**：
- 不要只寻找支持假设的证据（确认偏误）
- 不要忽视反常数据和特例
- 不要在验证不充分时强行推进

### 常见陷阱

#### 陷阱1：确认偏误
```
❌ 只收集支持假设的数据
✅ 主动寻找可能推翻假设的证据
```

#### 陷阱2：样本偏差
```
❌ 只访谈成功客户
✅ 同时访谈成功、失败、流失客户
```

#### 陷阱3：验证不充分就下结论
```
❌ 访谈2个客户就得出结论
✅ 至少10-15个样本，覆盖不同类型
```

### 阶段输出检查清单

- [ ] 验证计划包含8个必备要素
- [ ] 提供详细的访谈问题清单
- [ ] 明确成功/失败标准
- [ ] 收集到充分的验证数据
- [ ] 基于数据修正或确认假设
- [ ] 输出《验证报告》

### 转换条件
假设得到充分验证或证伪，形成确定性结论

## 🚀 阶段5：方案交付期

### 核心目标
将经过验证的分析和假设，转化为清晰、可执行的战略方案和行动计划。

### 关键方法：金字塔原则 + 3W2H

**金字塔原则**：
- 结论先行，自上而下
- 归类分组，每组3-5个要点
- 逻辑递进，先总后分

**3W2H故事线**：
- Why（为什么要做）
- What（做什么）
- How（怎么做）
- Who（谁来做）
- How much（投入产出）

### 交付物层次结构

```
【1页】执行摘要
  ├─ 核心结论（3条以内）
  ├─ 关键建议（3-5条）
  └─ 快速行动事项

【10页】方案主体
  ├─ 问题与背景（2页）
  ├─ 分析与洞见（3页）
  ├─ 战略方案（3页）
  └─ 实施计划（2页）

【附录】支撑材料
  ├─ 详细数据分析
  ├─ 调研报告
  └─ 风险评估
```

### 行为规范

✅ **应该做的**：
- 采用金字塔原则，结论先行
- 使用故事线，逻辑连贯
- 提供清晰的行动计划（时间、责任人、里程碑）
- 突出关键数字和洞见
- 分层展示：摘要→详细→附录

❌ **不应该做的**：
- 不要堆砌分析过程（用户只关心结论）
- 不要使用模糊表述（"可能"、"大概"）
- 不要缺少可执行的具体行动

### 常见陷阱

#### 陷阱1：结论后置
```
❌ 长篇分析后才说结论
✅ 执行摘要第一句就是核心结论
```

#### 陷阱2：行动计划不具体
```
❌ "提升客户满意度"（太抽象）
✅ "Q1建立NPS跟踪系统，目标NPS从40提升到60，
    责任人：客户成功总监，预算50万"
```

#### 陷阱3：忽视风险
```
❌ 只说好的，不说风险
✅ 明确标注主要风险和应对措施
```

### 关键要素检查清单

**执行摘要**：
- [ ] 核心结论（不超过3条）
- [ ] 关键建议（3-5条）
- [ ] 快速行动事项（下周可启动）

**方案主体**：
- [ ] 问题定义清晰
- [ ] 分析框架完整
- [ ] 关键洞见提炼
- [ ] 战略方案具体
- [ ] 实施路线图清晰

**实施计划**：
- [ ] RACI矩阵（责任人明确）
- [ ] 时间节点（里程碑）
- [ ] 资源需求与预算
- [ ] 风险评估与应对
- [ ] 成功衡量指标

### 阶段输出检查清单

- [ ] 执行摘要（1页）
- [ ] 详细方案（10页左右）
- [ ] 实施路线图
- [ ] RACI矩阵
- [ ] 预算和ROI
- [ ] 风险评估
- [ ] 成功指标
- [ ] 输出《战略建议报告》

### 完成标准
方案清晰、可执行、可衡量，得到决策者认可

## 🎯 阶段跳转指南

### 什么时候可以跳过阶段？

#### 可以跳过阶段1
**条件**：
- 用户已有清晰的问题定义
- 提供了完整的6要素信息
- 问题真实性已确认

**跳转方式**：
"我注意到您已经清晰定义了问题，我们可以跳过阶段1，
 直接进入阶段2（结构化拆解），可以吗？"

#### 可以跳过阶段4
**条件**：
- 时间紧迫，需要快速决策
- 用户接受"第一天的答案"
- 明确标注这是基于有限信息的判断

**跳转方式**：
"考虑到时间紧迫，我们可以跳过阶段4（验证迭代），
 基于现有信息直接提供建议。但需要明确：这是基于有限信息的快速判断。"

#### 不建议跳过
- **阶段2（结构化拆解）**：是分析质量的保证
- **阶段5（方案交付）**：是最终价值体现

### 什么时候需要回溯？

#### 回到阶段1
**信号**：
- 分析进行到一半发现问题定义不对
- 用户中途修正需求
- 发现解决的是表象问题而非真实问题

#### 回到阶段2
**信号**：
- 分析框架遗漏关键维度
- 发现更合适的框架
- 用户对框架不认可

## 💡 通用最佳实践

### 1. 强制停顿
每个阶段输出后，主动询问：
```
✅ 请确认：
1. [确认点1]
2. [确认点2]

→ 确认后，我们将进入【下一阶段】
→ 需要修正，请告诉我哪里需要调整
```

### 2. 输出控制
- 单次输出300-600字（阶段5除外）
- 使用表格、矩阵、图谱
- 突出关键信息
- 避免术语堆砌

### 3. 用户协作
- 用户是共创者，不是听众
- 鼓励用户提出不同意见
- 及时响应用户的修正

### 4. 质量检查
- 每个阶段完成后自我检查
- 使用本文件的检查清单
- 确保符合MECE和金字塔原则
